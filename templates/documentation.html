Theory

Session Types

A session type describes a communication protocol on an object — the legal sequences of method calls, branches, and selections that a client may perform. Instead of a flat interface, an object's type evolves as methods are called, enforcing protocol compliance at compile time.

Example: File Object

open . rec X . &{read: +{data: X, eof: close . end}}

Open the file, then repeatedly read: on data, loop back; on eof, close and terminate. The type ensures close is always called exactly once.

Grammar

Session types are defined by the following grammar:

S  ::=  &{ m₁ : S₁ , … , mₙ : Sₙ }    — branch (external choice)
     |  +{ l₁ : S₁ , … , lₙ : Sₙ }    — selection (internal choice)
     |  ( S₁ || S₂ )                    — parallel
     |  rec X . S                        — recursion
     |  X                                — variable
     |  end                              — terminated
     |  S₁ . S₂                          — sequencing

Constructors

&{…} — Branch

External choice. The environment (client) chooses which method to call. Each branch leads to a different continuation type.

+{…} — Selection

Internal choice. The object (server) decides the outcome. The client must handle all possibilities.

( S₁ || S₂ ) — Parallel

Concurrent access. Two execution paths run simultaneously on a shared object. The combined state space is the product lattice. This is the key novelty of this work.

rec X . S — Recursion

Looping protocols. The variable X marks the loop point. Well-formed recursive types must have an exit path.

S₁ . S₂ — Sequencing

Sequential composition. Syntactic sugar for a single-method branch: m . S is equivalent to &{m: S}.

end — Terminated

Protocol end. No further operations are allowed. Every well-formed session type must eventually reach end.

State Spaces

Given a session type S, we construct its state space L(S) — a directed graph where:

  • States are the reachable configurations of the protocol
  • Transitions are labeled edges (method calls, selections)
  • The initial state (top) is the protocol's entry point
  • The terminal state (bottom) corresponds to end

The reachability ordering defines a partial order on states: s₁ ≥ s₂ iff there is a path from s₁ to s₂.

Lattice Properties

A state space is a lattice (a reticulate) if and only if:

  1. There is a top element (initial state)
  2. There is a bottom element (terminal state)
  3. Every pair of states has a meet (greatest lower bound)
  4. Every pair of states has a join (least upper bound)

For cyclic state spaces (from recursion), we first quotient by SCCs to obtain an acyclic DAG, then check lattice properties on the quotient.

The Parallel Constructor

The ∥ constructor is the key novelty of this work. When two branches execute in parallel on a shared object:

L(S₁ ∥ S₂) = L(S₁) × L(S₂)

The product construction orders states componentwise. Crucially, the product of two lattices is always a lattice. This means that any well-formed session type using ∥ necessarily has a lattice state space.

Morphism Hierarchy

Between session-type state spaces, we define a hierarchy of structure-preserving maps:

  1. Isomorphism — bijective, order-preserving and reflecting.
  2. Embedding — injective, order-preserving and reflecting.
  3. Projection — surjective, order-preserving.
  4. Galois connection — an adjunction α(x) ≤ y ⇔ x ≤ γ(y).

Tutorials

Tutorial 1: Typechecking Object Usage with @Session

BICA Reborn is a Java annotation processor that enforces communication protocols at compile time. You annotate a class with @Session, write the protocol as a session type string, and the compiler checks that every client uses the object correctly.

Step 1: Add the Dependency

Clone the repository and install it to your local Maven cache:

Install locally

git clone https://github.com/zuacaldeira/SessionTypesResearch.git
cd SessionTypesResearch/bica
mvn install

Then add the dependency to your project:

pom.xml

<dependency>
    <groupId>com.bica</groupId>
    <artifactId>bica-reborn</artifactId>
    <version>0.1.0-SNAPSHOT</version>
    <scope>provided</scope>
</dependency>

Step 2: Annotate Your Class

FileHandle.java

import com.bica.reborn.annotation.Session;

@Session("open . &{read: close . end, write: close . end}")
public class FileHandle {
    // ...
}

This type says: first call open, then choose either read or write, then call close, then the protocol ends.

Step 3: Implement the Protocol

FileHandle.java (full)

import com.bica.reborn.annotation.Session;

@Session("open . &{read: close . end, write: close . end}")
public class FileHandle {
    private enum State { INITIAL, OPENED, PENDING_CLOSE, CLOSED }
    private State state = State.INITIAL;

    public void open() {
        requireState(State.INITIAL);
        state = State.OPENED;
    }

    public void read() {
        requireState(State.OPENED);
        state = State.PENDING_CLOSE;
    }

    public void write() {
        requireState(State.OPENED);
        state = State.PENDING_CLOSE;
    }

    public void close() {
        requireState(State.PENDING_CLOSE);
        state = State.CLOSED;
    }

    private void requireState(State expected) {
        if (state != expected)
            throw new IllegalStateException(
                "Expected " + expected + ", got " + state);
    }
}

Step 4: Write Correct Client Code

Client.java

public class Client {
    void use() {
        FileHandle f = new FileHandle();
        f.open();
        f.read();
        f.close();
        // Compiles — follows protocol: open → read → close → end
    }
}

Step 5: See Compile-Time Errors

BadClient.java

public class BadClient {
    void use() {
        FileHandle f = new FileHandle();
        f.read();   // ERROR: 'read' not enabled in state 0 (enabled: [open])
        f.close();
    }
}

The error message tells you exactly which methods are available in the current state.

Step 6: Add Concurrency Annotations

When your session type uses ∥, methods in concurrent branches need concurrency annotations:

  • @Shared — safe for concurrent access
  • @ReadOnly — read-only access
  • @Exclusive — exclusive access

Step 7: Generate Tests Automatically

BICA can generate JUnit 5 test suites from a session type covering valid paths, violations, and incomplete prefixes.

CLI usage

java -jar bica.jar --test-gen --class-name FileHandle \
    "open . &{read: close . end, write: close . end}"

You can also generate tests from the interactive tool by entering a class name.

Tutorial 2: Analyzing Session Types with the Reticulate CLI

Reticulate is a Python command-line tool that takes a session type string and performs the full analysis pipeline: parse, build the state space, check lattice properties, and optionally visualize the Hasse diagram.

Step 1: Install Reticulate

From source

git clone https://github.com/zuacaldeira/SessionTypesResearch.git
cd SessionTypesResearch/reticulate
python -m reticulate "end"

You only need Python 3.11+ and the standard library. The optional graphviz Python package is needed for Hasse diagrams.

Step 2: Your First Analysis

python -m reticulate "a . b . end"

Output:

Session type: a . b . end
States:       3
Transitions:  2
SCCs:         3
Lattice:      ✓ yes

Step 3: Branches and Selections

python -m reticulate "&{read: close . end, write: close . end}"

This gives a 4-state diamond: the initial state branches into read and write, both merging at close.

Step 4: Recursion

python -m reticulate "rec X . &{hasNext: +{TRUE: &{next: X}, FALSE: end}}"

The SCC quotient collapses cyclic states. The Recursion Lemma guarantees that if the quotient is a lattice, so is the original.

Step 5: The Parallel Constructor

python -m reticulate "(a.end || b.end)"

The state space is the product lattice L(S₁) × L(S₂), ordered componentwise.

Step 6: Visualize with --dot and --hasse

python -m reticulate --dot "a . b . end"
python -m reticulate --dot "a . b . end" | dot -Tpng -o diagram.png
python -m reticulate --hasse diagram "a . b . end"

Step 7: Real-World Protocols

python -m reticulate "connect . ehlo . rec X . &{mail: rcpt . data . +{OK: X, ERR: X}, quit: end}"

The project includes 34 benchmark protocols. Browse them in the benchmarks gallery or paste any protocol into the interactive analyzer.

FAQ

What is a session type?

A session type describes the communication protocol governing interaction with an object — the legal sequences of method calls, branches, and selections a client may perform. Unlike a flat interface, a session type makes the object's type evolve as methods are called, enforcing protocol compliance statically.

What is a state space?

A state space L(S) is the labeled transition system obtained by "executing" a session type. Each state represents a protocol stage, each transition a permitted action. It has a unique initial state (top) and terminal state (bottom). The reachability ordering gives it the structure of a bounded lattice (after SCC quotient).

What is a bounded lattice?

A bounded lattice is a partially ordered set where every pair has a meet (greatest lower bound) and join (least upper bound), plus top and bottom elements. In session types: top = initial state, bottom = terminal state, meet = convergence point, join = divergence point.

What is the parallel constructor?

The parallel constructor S₁ ∥ S₂ models two sub-protocols executing concurrently on a shared object. Its state space is the product of the two components. This is the key novelty — the product of two lattices is always a lattice, making lattice structure necessary.

What is a reticulate?

A reticulate is the bounded lattice formed by the state space of a well-formed session type, after quotienting by SCCs. The Reticulate Theorem proves every well-formed session type produces one — it is a guaranteed structural property.

What is an SCC quotient?

An SCC quotient collapses every strongly connected component into a single node, turning a cyclic graph into a DAG. In session types, cycles from recursion are collapsed, restoring antisymmetry for the partial order.

What is top absorption?

Top absorption is the key lemma for recursion. It states: if you collapse an upward-closed set containing top into a single element, the result is still a bounded lattice. This has been mechanically verified in Lean 4.

What is WF-Par?

WF-Par is the well-formedness condition for the parallel constructor. It requires both branches to be terminating, well-formed, variable-disjoint, and free of nested ∥.

What is a morphism between session types?

A morphism is a structure-preserving map between state spaces. The hierarchy: homomorphism (order-preserving), projection (surjective), embedding (injective + reflecting), isomorphism (bijective embedding). Additionally, Galois connections capture approximation relationships.

Why do session type state spaces form lattices?

Because every constructor preserves lattice structure, and the proof goes by structural induction: end is trivial; sequencing adds a maximum; branch/selection create joins and meets; recursion is absorbed by SCC quotient; parallel takes the product. Since every constructor preserves the property and the base case has it, every well-formed session type has it.
