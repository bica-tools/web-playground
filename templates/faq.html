FAQ

Frequently asked questions about session types and reticulates.

What is a meet in a session type?

In a session type's state space, the meet of two states s₁ and s₂ is the state s₁ ∧ s₂ that is the greatest lower bound — the most advanced state reachable from both s₁ and s₂.

It represents the latest point where two execution paths converge: the most advanced common "future" of both states.

For states in different branches of a choice (& or +), the meet is always the terminal state q⊥, since the only place both branches eventually reach is the end of the protocol.

Example

In &{m: a.end, n: b.end}, after choosing m you are in state q₁; after n, in state q₂. These branches are disjoint — their meet is q⊥ (the terminal state), the only state reachable from both.

What is a join in a session type?

The join of two states s₁ and s₂ is the state s₁ ∨ s₂ that is the least upper bound — the earliest state from which both s₁ and s₂ can be reached.

It represents the latest point before two execution paths diverge: the most advanced common "past" of both states.

For states in different branches of a choice (& or +), the join is always the initial state q⊤, since the only state that can reach into both branches is the entry point of the choice.

Example

In &{m: a.end, n: b.end}, states q₁ (after m) and q₂ (after n) are in disjoint branches. The only state that reaches both is q₀ (the initial state) — so q₁ ∨ q₂ = q₀ = q⊤.

What is a reticulate?

A reticulate is the bounded lattice formed by the state space of a well-formed session type, after quotienting by strongly connected components.

The name comes from the lattice-like "net" structure that emerges: every pair of states has a well-defined meet (convergence point) and join (divergence point), with a unique top (initial state, q⊤) and bottom (terminal state, q⊥).

The Reticulate Theorem proves that every well-formed session type produces one — it is not something you check for, it is a guaranteed structural property of session types. The proof is constructive, by structural induction: end gives a trivial lattice; sequencing adds a new maximum; branch and selection create joins and meets; recursion is absorbed by the SCC quotient; and parallel composition yields a product lattice.

What is the parallel constructor?

The parallel constructor S₁ ∥ S₂ allows two sub-protocols to execute concurrently on the same shared object. Its state space is the product of the two component state spaces:

L(S₁ ∥ S₂) = L(S₁) × L(S₂)

Transitions are componentwise — each step advances exactly one component — and the ordering is also componentwise: (s₁, s₂) ≥ (t₁, t₂) iff s₁ ≥ t₁ and s₂ ≥ t₂.

This is the key novelty of the work — it is where lattice structure becomes necessary rather than merely convenient, because the product of two lattices is always a lattice.

Well-formedness (WF-Par) requires both branches to be terminating, variable-disjoint, and free of nested ∥, ensuring the branches always synchronise at a common terminal state.

Example

(a . b . end || c . d . end) creates a 3 × 3 = 9 state product lattice. The initial state (0,0) is at the top; the terminal (2,2) at the bottom. Every interleaving of a, b with c, d is a valid path through the grid.

What is an SCC quotient?

An SCC quotient collapses every strongly connected component (a maximal set of mutually reachable states) into a single node, turning a cyclic graph into a DAG (directed acyclic graph).

In session types, cycles arise from recursion — rec X . S creates back-edges that make some states mutually reachable. The quotient merges those cyclic states into one equivalence class, restoring the antisymmetry needed for a partial order.

The Reticulate Theorem works on this quotient: it is the SCC quotient that forms a bounded lattice, not the raw state space (which is only a preorder due to cycles).

Example

The file object open . rec X . &{read: +{data: X, eof: close . end}} has states q₁ (read) and q₂ (select) forming a cycle via data. The SCC quotient collapses {q₁, q₂} into a single node, yielding a 4-element chain — trivially a lattice.

What is top absorption?

Top absorption is the key lemma that handles recursion in the Reticulate Theorem. It states: if you take a bounded lattice and collapse an upward-closed set containing ⊤ into a single top element, the result is still a bounded lattice.

This is exactly what happens when recursion adds back-edges — states that become mutually reachable with the initial state (q⊤) get absorbed into its SCC. The absorbed set is upward-closed because if a state is on a cycle through q⊤ and another state is above it, that state is also on the cycle.

The proof works in two parts:

  • Meets are preserved — the surviving states form a downward-closed set (order-ideal), so meets of survivors stay among the survivors.
  • Joins — if the join of two survivors lands in the absorbed set, it simply becomes ⊤. No non-⊤ upper bound can exist among the survivors (it would contradict downward-closure).

This lemma has been mechanically verified in Lean 4 with Mathlib (~480 lines, zero sorry).

What is WF-Par?

WF-Par is the well-formedness condition for the parallel constructor S₁ ∥ S₂. It requires four things:

  1. Both branches are terminating — every maximal path reaches end, so the branches always synchronise at a common terminal state.
  2. Both branches are individually well-formed.
  3. Variable-disjoint — no recursion variable bound in one branch is free in the other, keeping the branches independent.
  4. No nested ∥ — neither branch contains another parallel constructor.

These conditions are sufficient for lattice structure (a corollary of the Reticulate Theorem) and make the product construction well-defined. They are checked by the check_wf_parallel() function in both Reticulate and BICA Reborn.

Example

(rec X . &{a: X}) || (b . end) violates WF-Par: the left branch is not terminating (it loops forever with no exit path to end).

Why do session type state spaces form lattices?

Because every session type constructor preserves lattice structure, and the proof goes by structural induction:

  1. end is a one-element lattice (trivially).
  2. Sequencing (m . S) adds a new maximum — still a lattice.
  3. Branch and selection create a fresh top (the join of cross-branch states) and share a common bottom (the meet), producing a diamond-like structure where every pair has meets and joins.
  4. Recursion creates cycles, but the SCC quotient absorbs them into the top via top absorption — still a lattice.
  5. Parallel takes the product of two lattices, which is always a lattice with componentwise meets and joins.

Since every constructor preserves the property and the base case has it, every well-formed session type has it. The lattice structure is not imposed — it is an inevitable consequence of how protocols are built from these constructors.
