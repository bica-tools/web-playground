FAQ

Frequently asked questions about session types and reticulates.

What is a session type?

A session type describes the communication protocol governing interaction with an object — the legal sequences of method calls, branches, and selections a client may perform.

Unlike a flat interface where any method can be called at any time, a session type makes the object's type evolve as methods are called: after open, you can read; after read returns eof, you must close. The type enforces protocol compliance statically (at compile time), preventing errors like reading a closed file or closing a file twice.

Session types are built from constructors: branch (&), selection (+), parallel (∥), recursion (rec X . S), sequencing (.), and termination (end).

Example: File Object

open . rec X . &{read: +{data: X, eof: close . end}}

Open the file, then repeatedly read: on data, loop back; on eof, close and terminate. The type guarantees close is called exactly once.

What is a state space?

A state space L(S) is the labeled transition system obtained by "executing" a session type S. Each state represents a protocol stage, and each transition a permitted action (method call or selection label).

It has a unique initial state q⊤ (the protocol entry point) and a unique terminal state q⊥ (corresponding to end). Formally, a state space is a tuple L(S) = (Q, Σ, δ, q⊤, q⊥).

The construction is inductive: end gives a single state; sequencing chains states; branch/selection fan out from a fresh initial to each sub-space; recursion adds back-edges creating cycles; and parallel builds the product of two state spaces.

The reachability ordering on states (s₁ ≥ s₂ iff path from s₁ to s₂) gives the state space the structure of a poset — and after SCC quotient, a bounded lattice.

What is branch and selection?

Branch (&{m₁: S₁, …, mₙ: Sₙ}) and selection (+{l₁: S₁, …, lₙ: Sₙ}) are the two choice constructors.

Branch is external choice — the client (environment) decides which method to call. Selection is internal choice — the object (server) decides the outcome, and the client must handle all possibilities.

Despite this type-theoretic duality, they produce identical state-space structure: a fresh initial state fans out to each sub-state-space, and all sub-state-spaces share a common terminal state. The initial state is the join of all cross-branch states; the terminal is their meet.

This is why the lattice proof treats them uniformly as C{a₁: S₁, …, aₙ: Sₙ}.

Example

&{read: end, write: end} — the client chooses to read or write.
+{OK: end, ERROR: end} — the server returns OK or ERROR, the client must handle both.

What is the end constructor?

end is the termination constructor — it means the protocol is finished and no more method calls are permitted on the object.

Its state space is the simplest possible: a single state with no transitions. That single state is simultaneously the initial state q⊤ and the terminal state q⊥. A one-element set is trivially a bounded lattice, making end the base case of the Reticulate Theorem's induction.

Every well-formed session type must eventually reach end along every maximal path — this is the termination property required by WF-Par for parallel branches, and it ensures the state space always has a well-defined bottom element.

Example

close . end — call close, then the protocol terminates. After end, any further method call is a protocol violation.

What is sequencing in session types?

Sequencing (m . S) is the simplest session type constructor — it says "call method m, then continue with protocol S". It chains method calls into a strict order.

In the state space, sequencing adds a new state above the sub-protocol's state space, connected by a single transition labeled m. Formally, if L(S) has initial state q⊤, then L(m . S) introduces a fresh state q that becomes the new q⊤, with a transition q ⟶ q⊤(S) labeled m.

For the lattice proof, sequencing is the easiest case: adding a new maximum to a bounded lattice always yields a bounded lattice — the new state is above everything, so it is trivially the join of any pair involving itself, and all existing meets and joins are unchanged.

Syntactically, m . S is sugar for a single-arm branch: &{m: S}. The parser desugars it automatically.

Example

open . read . close . end creates a 4-state chain: q₀ ⟶ q₁ ⟶ q₂ ⟶ q₃, where q₀ = q⊤ and q₃ = q⊥. Every chain is trivially a lattice.

What is recursion in session types?

Recursion (rec X . S) allows a session type to loop back to an earlier protocol stage. The variable X marks the loop point, and any occurrence of X inside S means "go back to the beginning of the rec".

In the state space, recursion creates back-edges — transitions from deeper states back to the entry state of the recursive body. This makes some states mutually reachable, forming cycles in the transition graph.

These cycles break antisymmetry (if s₁ ≥ s₂ and s₂ ≥ s₁, we need s₁ = s₂ for a partial order). The solution is the SCC quotient: collapse every cycle into a single equivalence class. The Top Absorption Lemma then guarantees the quotient is still a bounded lattice.

For well-formedness, recursive types used inside ∥ must be terminating — every maximal path must have an exit to end, not just loop forever.

Example

rec X . &{next: X, stop: end} — repeatedly call next to loop, or stop to terminate. The back-edge from next to the entry creates a cycle; the SCC quotient collapses it into a 2-element chain: [loop] > [end].

What is a bounded lattice?

A bounded lattice is a partially ordered set (poset) where:

  1. Every pair of elements has a meet (greatest lower bound, ∧)
  2. Every pair of elements has a join (least upper bound, ∨)
  3. There is a top element ⊤ (greater than or equal to everything)
  4. There is a bottom element ⊥ (less than or equal to everything)

In the context of session types: the top is the initial state, the bottom is the terminal state, the meet of two states is where their execution paths converge, and the join is where they diverge.

The "bounded" part matters — it guarantees the protocol has a well-defined entry point (⊤) and exit point (⊥), which is exactly what well-formedness ensures.

What is a meet in a session type?

In a session type's state space, the meet of two states s₁ and s₂ is the state s₁ ∧ s₂ that is the greatest lower bound — the most advanced state reachable from both s₁ and s₂.

It represents the latest point where two execution paths converge: the most advanced common "future" of both states.

For states in different branches of a choice (& or +), the meet is always the terminal state q⊥, since the only place both branches eventually reach is the end of the protocol.

Example

In &{m: a.end, n: b.end}, after choosing m you are in state q₁; after n, in state q₂. These branches are disjoint — their meet is q⊥ (the terminal state), the only state reachable from both.

What is a join in a session type?

The join of two states s₁ and s₂ is the state s₁ ∨ s₂ that is the least upper bound — the earliest state from which both s₁ and s₂ can be reached.

It represents the latest point before two execution paths diverge: the most advanced common "past" of both states.

For states in different branches of a choice (& or +), the join is always the initial state q⊤, since the only state that can reach into both branches is the entry point of the choice.

Example

In &{m: a.end, n: b.end}, states q₁ (after m) and q₂ (after n) are in disjoint branches. The only state that reaches both is q₀ (the initial state) — so q₁ ∨ q₂ = q₀ = q⊤.

What is a reticulate?

A reticulate is the bounded lattice formed by the state space of a well-formed session type, after quotienting by strongly connected components.

The name comes from the lattice-like "net" structure that emerges: every pair of states has a well-defined meet (convergence point) and join (divergence point), with a unique top (initial state, q⊤) and bottom (terminal state, q⊥).

The Reticulate Theorem proves that every well-formed session type produces one — it is not something you check for, it is a guaranteed structural property of session types. The proof is constructive, by structural induction: end gives a trivial lattice; sequencing adds a new maximum; branch and selection create joins and meets; recursion is absorbed by the SCC quotient; and parallel composition yields a product lattice.

What is the parallel constructor?

The parallel constructor S₁ ∥ S₂ allows two sub-protocols to execute concurrently on the same shared object. Its state space is the product of the two component state spaces:

L(S₁ ∥ S₂) = L(S₁) × L(S₂)

Transitions are componentwise — each step advances exactly one component — and the ordering is also componentwise: (s₁, s₂) ≥ (t₁, t₂) iff s₁ ≥ t₁ and s₂ ≥ t₂.

This is the key novelty of the work — it is where lattice structure becomes necessary rather than merely convenient, because the product of two lattices is always a lattice.

Well-formedness (WF-Par) requires both branches to be terminating, variable-disjoint, and free of nested ∥, ensuring the branches always synchronise at a common terminal state.

Example

(a . b . end || c . d . end) creates a 3 × 3 = 9 state product lattice. The initial state (0,0) is at the top; the terminal (2,2) at the bottom. Every interleaving of a, b with c, d is a valid path through the grid.

What is a Hasse diagram?

A Hasse diagram is the standard way to draw a partially ordered set (poset). It shows elements as nodes and draws an edge from a down to b only when a covers b — meaning a > b with nothing in between.

By omitting transitive edges (if a > b > c, don't draw a → c), the diagram stays clean and readable. The full ordering can always be recovered: a ≥ b iff there is a downward path from a to b.

In Reticulate, the Hasse diagram of a session type's state space is the primary visualization. The top (initial state, q⊤) sits at the top of the diagram, the bottom (terminal state, q⊥) at the bottom, and transitions are drawn as labeled edges flowing downward. Edges are labeled with method names or selection labels.

The tool colours the top node blue, the bottom node red, and highlights lattice counterexamples (missing meets or joins) when they exist. SCC-collapsed nodes are shown as grouped clusters.

What is an SCC quotient?

An SCC quotient collapses every strongly connected component (a maximal set of mutually reachable states) into a single node, turning a cyclic graph into a DAG (directed acyclic graph).

In session types, cycles arise from recursion — rec X . S creates back-edges that make some states mutually reachable. The quotient merges those cyclic states into one equivalence class, restoring the antisymmetry needed for a partial order.

The Reticulate Theorem works on this quotient: it is the SCC quotient that forms a bounded lattice, not the raw state space (which is only a preorder due to cycles).

Example

The file object open . rec X . &{read: +{data: X, eof: close . end}} has states q₁ (read) and q₂ (select) forming a cycle via data. The SCC quotient collapses {q₁, q₂} into a single node, yielding a 4-element chain — trivially a lattice.

What is a protocol violation?

A protocol violation occurs when a client calls a method that is not permitted in the object's current protocol state. The method may exist on the object, but the session type says "not now".

In the state space, a violation means attempting a transition label that has no outgoing edge from the current state. For example, if the protocol is in state q₁ and only transition read is available, calling close is a violation.

Session type systems detect violations statically (at compile time) by tracking the object's type as it evolves through method calls. The type system rejects programs that could ever reach a violating call. This is the core value proposition: bugs like reading a closed file, double-closing, or sending on a finished connection are caught before the program runs.

Reticulate's test generation produces three kinds of test cases from a session type:

  • Valid paths — complete traces from q⊤ to q⊥ (should succeed)
  • Violations — a valid prefix followed by a disabled method call (should throw)
  • Incomplete prefixes — a valid prefix that stops before q⊥ (resource leak)

Example

For open . read . close . end, calling read before open is a violation — the initial state only permits open. Calling open after read is also a violation — that state only permits close.

What is termination in session types?

A session type is terminating if every maximal execution path eventually reaches end. There is no way to get stuck in an infinite loop with no exit.

This is not the same as forbidding recursion — recursive types can be terminating as long as every recursive loop has at least one branch that leads to end. The check is decidable: for each rec X . S, verify that the body S has an exit path that does not go through X.

Termination matters in two places:

  • WF-Par requires both branches of S₁ ∥ S₂ to be terminating, ensuring the concurrent branches always synchronise at a common terminal state.
  • State-space finiteness — a non-terminating type inside a parallel can produce an infinite product, which defeats analysis.

Outside of parallel, non-terminating types like rec X . a . X are perfectly valid — they model servers or event loops that run forever. The SCC quotient handles the resulting cycles.

Examples

rec X . &{next: X, stop: end} — terminating: the stop branch exits.
rec X . a . X — not terminating: every path loops back to X with no exit.

What is a morphism between session types?

A morphism between two session type state spaces is a function f : L(S₁) → L(S₂) that preserves the ordering structure. If s₁ ≥ s₂ in the source, then f(s₁) ≥ f(s₂) in the target — the map respects the reachability relation.

Reticulate classifies morphisms into a hierarchy of increasing strength:

  1. Homomorphism — order-preserving: if a ≥ b then f(a) ≥ f(b). The weakest useful notion.
  2. Projection — order-preserving and surjective: every state in the target is hit. The source "covers" the target protocol.
  3. Embedding — order-preserving and order-reflecting (injective on the ordering): f(a) ≥ f(b) implies a ≥ b. The source "sits inside" the target faithfully.
  4. Isomorphism — bijective embedding. The two state spaces are structurally identical.

Additionally, a Galois connection is an adjoint pair (α, γ) where α(x) ≤ y iff x ≤ γ(y), capturing an "approximation" relationship between protocols.

Morphisms let you answer questions like: "Is protocol A a refinement of protocol B?" or "Can a client built for protocol A safely interact with an object following protocol B?"

What is a Galois connection?

A Galois connection between two posets (or lattices) is a pair of monotone functions (α, γ) satisfying the adjunction law:

α(x) ≤ y ⇔ x ≤ γ(y)

Here α is the abstraction (left adjoint) and γ is the concretisation (right adjoint). Together they express a "best approximation" relationship: α maps into a simpler domain, and γ maps back, with the round-trips satisfying x ≤ γ(α(x)) and α(γ(y)) ≤ y.

In session types, a Galois connection between L(S₁) and L(S₂) means that one protocol is a sound approximation of the other — the abstraction may lose detail, but it never introduces false behaviour. This is weaker than an embedding (which preserves structure exactly) but stronger than a plain homomorphism.

Galois connections are central to abstract interpretation and program analysis, where they formalise the relationship between concrete and abstract domains. In our setting, they connect protocol refinement to the lattice-theoretic framework.

Example

If protocol S₁ has states {q₀, q₁, q₂, q₃} and S₂ collapses q₁ and q₂ into a single state, the abstraction α maps both to the collapsed state, and the concretisation γ maps back to their join. The adjunction law ensures this approximation is consistent.

What is top absorption?

Top absorption is the key lemma that handles recursion in the Reticulate Theorem. It states: if you take a bounded lattice and collapse an upward-closed set containing ⊤ into a single top element, the result is still a bounded lattice.

This is exactly what happens when recursion adds back-edges — states that become mutually reachable with the initial state (q⊤) get absorbed into its SCC. The absorbed set is upward-closed because if a state is on a cycle through q⊤ and another state is above it, that state is also on the cycle.

The proof works in two parts:

  • Meets are preserved — the surviving states form a downward-closed set (order-ideal), so meets of survivors stay among the survivors.
  • Joins — if the join of two survivors lands in the absorbed set, it simply becomes ⊤. No non-⊤ upper bound can exist among the survivors (it would contradict downward-closure).

What is WF-Par?

WF-Par is the well-formedness condition for the parallel constructor S₁ ∥ S₂. It requires four things:

  1. Both branches are terminating — every maximal path reaches end, so the branches always synchronise at a common terminal state.
  2. Both branches are individually well-formed.
  3. Variable-disjoint — no recursion variable bound in one branch is free in the other, keeping the branches independent.
  4. No nested ∥ — neither branch contains another parallel constructor.

These conditions are sufficient for lattice structure (a corollary of the Reticulate Theorem) and make the product construction well-defined. They are checked by the check_wf_parallel() function in both Reticulate and BICA Reborn.

Example

(rec X . &{a: X}) || (b . end) violates WF-Par: the left branch is not terminating (it loops forever with no exit path to end).

Why do session type state spaces form lattices?

Because every session type constructor preserves lattice structure, and the proof goes by structural induction:

  1. end is a one-element lattice (trivially).
  2. Sequencing (m . S) adds a new maximum — still a lattice.
  3. Branch and selection create a fresh top (the join of cross-branch states) and share a common bottom (the meet), producing a diamond-like structure where every pair has meets and joins.
  4. Recursion creates cycles, but the SCC quotient absorbs them into the top via top absorption — still a lattice.
  5. Parallel takes the product of two lattices, which is always a lattice with componentwise meets and joins.

Since every constructor preserves the property and the base case has it, every well-formed session type has it. The lattice structure is not imposed — it is an inevitable consequence of how protocols are built from these constructors.
