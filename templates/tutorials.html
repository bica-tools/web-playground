Tutorial 1: Typechecking Object Usage with @Session

BICA Reborn is a Java annotation processor that enforces communication protocols at compile time. You annotate a class with @Session, write the protocol as a session type string, and the compiler checks that every client uses the object correctly — no extra tools needed.

In this tutorial you will annotate a FileHandle class, write correct and incorrect client code, add concurrency annotations, and generate JUnit 5 tests automatically.

Step 1: Add the Dependency

BICA Reborn is not yet published to Maven Central. Clone the repository and install it to your local Maven cache first:

Install locally

git clone https://github.com/zuacaldeira/SessionTypesResearch.git
cd SessionTypesResearch/bica
mvn install

Then add the dependency to your project:

pom.xml

<dependency>
    <groupId>com.bica</groupId>
    <artifactId>bica-reborn</artifactId>
    <version>0.1.0-SNAPSHOT</version>
    <scope>provided</scope>
</dependency>

The <scope>provided</scope> is correct because the annotations use SOURCE retention — they are only needed during compilation, not at runtime.

Step 2: Annotate Your Class

Declare the protocol as a session type string on the class. This tells the compiler what sequences of method calls are legal.

FileHandle.java

import com.bica.reborn.annotation.Session;

@Session("open . &{read: close . end, write: close . end}")
public class FileHandle {
    // ...
}

This type says: first call open, then choose either read or write (branch), then call close, then the protocol ends. No other call sequences are allowed.

Step 3: Implement the Protocol

Implement the class with a state machine that enforces the protocol at runtime. The annotation processor validates the static call sequences; the runtime guards catch dynamic violations (e.g., from reflection).

FileHandle.java (full)

import com.bica.reborn.annotation.Session;

@Session("open . &{read: close . end, write: close . end}")
public class FileHandle {
    private enum State { INITIAL, OPENED, PENDING_CLOSE, CLOSED }
    private State state = State.INITIAL;

    public void open() {
        requireState(State.INITIAL);
        state = State.OPENED;
    }

    public void read() {
        requireState(State.OPENED);
        state = State.PENDING_CLOSE;
    }

    public void write() {
        requireState(State.OPENED);
        state = State.PENDING_CLOSE;
    }

    public void close() {
        requireState(State.PENDING_CLOSE);
        state = State.CLOSED;
    }

    private void requireState(State expected) {
        if (state != expected)
            throw new IllegalStateException(
                "Expected " + expected + ", got " + state);
    }
}

Step 4: Write Correct Client Code

A client that follows the protocol compiles without errors. The annotation processor traces the call sequence through the state space and verifies that it forms a valid path from top (initial) to bottom (end).

Client.java

public class Client {
    void use() {
        FileHandle f = new FileHandle();
        f.open();
        f.read();
        f.close();
        // Compiles — follows protocol: open → read → close → end
    }
}

Step 5: See Compile-Time Errors

A client that violates the protocol gets a compiler error. No need to run the program to discover the bug.

BadClient.java

public class BadClient {
    void use() {
        FileHandle f = new FileHandle();
        f.read();   // ERROR: 'read' not enabled in state 0 (enabled: [open])
        f.close();
    }
}

The error message tells you exactly which methods are available in the current state. Here, state 0 is the initial state where only open is enabled — calling read before open is a protocol violation.

Step 6: Add Concurrency Annotations

When your session type uses the ∥ (parallel) constructor, methods in concurrent branches need concurrency annotations to ensure thread safety. BICA provides three:

  • @Shared — safe for concurrent access (e.g., immutable reads)
  • @ReadOnly — read-only access, compatible with other readers
  • @Exclusive — exclusive access, incompatible with any concurrent call

SharedFile.java — invalid

import com.bica.reborn.annotation.*;

@Session("(read . end || write . end)")
public class SharedFile {
    @ReadOnly
    public void read() { /* ... */ }

    @Exclusive
    public void write() { /* ... */ }
    // ERROR: read (READ_ONLY) and write (EXCLUSIVE) in parallel — not compatible
}

The processor detects that read and write run concurrently but have incompatible concurrency levels. Fix: either make both @Shared, restructure the protocol to sequence them, or use synchronization.

Step 7: Generate Tests Automatically

BICA can generate JUnit 5 test suites from a session type. The generated tests cover three categories:

  • Valid paths — every top-to-bottom path through the protocol
  • Violations — calling a disabled method at each reachable state
  • Incomplete prefixes — stopping before reaching end

CLI usage

java -jar bica.jar --test-gen --class-name FileHandle \
    "open . &{read: close . end, write: close . end}"

This outputs a complete JUnit 5 test class. You can also generate tests from the interactive tool by entering a class name.

Tutorial 2: Analyzing Session Types with the Reticulate CLI

Reticulate is a Python command-line tool that takes a session type string and performs the full analysis pipeline: parse the type, build the state space, check whether it forms a lattice, and optionally visualize the Hasse diagram. It reports the number of states, transitions, strongly connected components (SCCs), and a lattice verdict.

Three output modes are available: a text summary (default), DOT source for Graphviz (--dot), and a rendered Hasse diagram image (--hasse).

Step 1: Install Reticulate

Install from PyPI or run directly from source:

From PyPI

pip install reticulate

From source

git clone https://github.com/zuacaldeira/SessionTypesResearch.git
cd SessionTypesResearch/reticulate
python -m reticulate "end"

If running from source, you only need Python 3.11+ and the standard library. The optional graphviz Python package is needed only for rendering Hasse diagrams (--hasse).

Step 2: Your First Analysis

Start with the simplest non-trivial session type — a linear sequence of two method calls: 

python -m reticulate "a . b . end"

The output looks like:

Session type: a . b . end
States:       3
Transitions:  2
SCCs:         3
Lattice:      ✓ yes

The state space has three states (one per position: before a, between a and b, and after b at end) connected by two transitions. All three states are in separate SCCs (no cycles), and the space forms a lattice — in this case a simple chain.

Step 3: Branches and Selections

The branch constructor &{...} models external choice — the environment picks which method to call. Try a file handle protocol with a read/write branch:

Branch (external choice)

python -m reticulate "&{read: close . end, write: close . end}"

This gives 4 states and 4 transitions forming a diamond shape: the initial state branches into read and write, and both paths merge at close before reaching end.

The selection constructor +{...} models internal choice — the object decides which branch to take:

Selection (internal choice)

python -m reticulate "+{OK: data . end, ERROR: retry . end}"

Structurally, branch and selection produce similar state spaces. The distinction matters for subtyping and duality, but both form lattices.

Step 4: Recursion

The rec X . S constructor models repeating protocols. The classic example is a Java Iterator: 

python -m reticulate "rec X . &{hasNext: +{TRUE: &{next: X}, FALSE: end}}"

This protocol says: call hasNext, then the object responds with TRUE (call next and repeat) or FALSE (done). The state space contains a cycle from the recursive variable X.

To handle cycles, Reticulate computes the SCC quotient — it collapses strongly connected components into single nodes and checks the lattice property on the resulting DAG. The Recursion Lemma guarantees that if the quotient is a lattice, so is the original.

Step 5: The Parallel Constructor

The ∥ (parallel) constructor is the key novelty of the reticulate framework. It models two concurrent execution paths on a shared object:

Simple parallel

python -m reticulate "(a.end || b.end)"

The state space is the product lattice L(S1) × L(S2), ordered componentwise. For (a.end || b.end), each branch has 2 states, so the product has 2 × 2 = 4 states.

A more realistic example — a simplified Two-Buyer protocol:

Two-Buyer protocol

python -m reticulate "lookup . getPrice . (proposeA . end || proposeB . +{ACCEPT: pay . end, REJECT: end})"

After the sequential prefix (lookup, getPrice), buyer A and buyer B proceed concurrently. Buyer B makes a selection (accept or reject). The product lattice captures all valid interleavings.

Step 6: Visualize with --dot and --hasse

Reticulate can output the state space as a Graphviz DOT graph. Use --dot to print DOT source to stdout:

DOT output

python -m reticulate --dot "a . b . end"

Pipe the output to Graphviz to render an image:

python -m reticulate --dot "a . b . end" | dot -Tpng -o diagram.png

Or use --hasse to render directly (requires the graphviz Python package):

Rendered Hasse diagram

python -m reticulate --hasse diagram "a . b . end"

Additional formatting flags:

# SVG format, custom title, hide edge labels
python -m reticulate --hasse diagram --fmt svg --title "My Protocol" --no-edge-labels "a . b . end"

# Hide state labels too
python -m reticulate --hasse diagram --no-labels "a . b . end"

Step 7: Real-World Protocols

Reticulate handles complex, real-world protocols. Try the SMTP email protocol:

SMTP

python -m reticulate "connect . ehlo . rec X . &{mail: rcpt . data . +{OK: X, ERR: X}, quit: end}"

This models the SMTP session: connect, send EHLO, then loop over mail/rcpt/data transactions (each ending in OK or ERR to continue) until quit. The state space has 7 states and forms a lattice.
